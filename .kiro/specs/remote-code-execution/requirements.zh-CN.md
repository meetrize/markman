# 需求文档

## 简介

Velotype 是一个专注于快速、无干扰写作和编码的轻量级代码编辑器。目前,代码块执行发生在用户机器本地,这需要在本地安装语言运行时和依赖项。这会增加应用程序的资源占用,并可能减慢响应速度。

此功能允许在远程服务器上执行代码块,而不是在本地执行,同时保持应用程序在大小、内存使用和响应时间方面的轻量级。用户应该能够为每个 Markdown 文档指定服务器配置,避免重复认证,并选择使用外部终端集成或轻量级内置解决方案。

## 术语表

- **Velotype**: 代码编辑器应用程序
- **代码块**: Markdown 文档中包含可执行代码的部分(带语言标识符的带边框代码块)
- **远程服务器**: 远程执行环境,在远程服务器上运行代码片段而不是在本地机器上
- **执行上下文**: 代码执行的环境(本地与远程)
- **执行会话**: 共享相同服务器连接和状态的一系列代码执行
- **前置元数据(Frontmatter)**: Markdown 文档开头的元数据部分
- **执行管理器**: 处理远程代码执行编排的组件
- **认证管理器**: 安全地存储和检索认证凭据的组件
- **连接池**: 管理到远程服务器的可重用连接的组件
- **上下文管理器**: 在多次执行之间维护执行状态的组件
- **输出处理器**: 路由执行结果到显示(内置或外部终端)的组件
- **服务器配置**: 包含主机名、端口、协议、认证方法和超时设置的配置对象

## 需求

### 需求 1: 远程代码执行能力

**用户故事:** 作为用户,我希望在远程服务器上执行代码块,这样我就无需在本地安装语言运行时和依赖项。

#### 验收标准

1. WHEN 代码块被执行 AND 远程执行已启用,THE Velotype 应将代码发送到配置的远程服务器进行执行
2. WHILE 远程执行进行中,THE Velotype 应显示执行状态和进度
3. WHEN 远程执行完成,THE Velotype 应在编辑器中显示输出、退出码和执行时间
4. IF 远程服务器不可用或无法访问,THEN THE Velotype 应显示清晰的错误消息
5. WHERE 未配置服务器,THE Velotype 应默认使用本地执行

### 需求 2: 轻量级应用要求

**用户故事:** 作为用户,我希望远程代码执行对应用程序性能的影响最小,这样 Velotype 保持轻量和响应迅速。

#### 验收标准

1. THE 从远程执行功能增加的总应用程序大小应小于 5MB
2. WHILE 执行远程代码时,THE Velotype 应使用少于 100MB 的额外内存
3. WHEN 发送代码进行执行时,THE Velotype 应在 500ms 内完成 10KB 以下的有效负载请求
4. WHERE 代码块超过 100KB,THE Velotype 应在发送到远程服务器之前警告用户

### 需求 3: 每文档服务器配置

**用户故事:** 作为用户,我希望为每个 Markdown 文档指定服务器配置,这样不同文档可以使用不同的执行环境。

#### 验收标准

1. WHEN Markdown 文档在前置元数据中包含服务器配置,THE Velotype 应使用该配置进行代码块执行
2. WHERE 文档未包含服务器配置,THE Velotype 应使用默认全局服务器配置
3. IF 前置元数据中的服务器配置无效,THEN THE Velotype 应使用全局默认配置并通知用户
4. THE 服务器配置格式应支持:主机名、端口、协议(http/https)、认证方法和执行超时

### 需求 4: 认证管理

**用户故事:** 作为用户,我希望认证一次并重复使用我的凭据,这样我就无需为每次代码执行重新输入凭据。

#### 验收标准

1. WHEN 用户为远程服务器提供认证凭据,THE Velotype 应安全地存储它们
2. WHILE 对配置的服务器执行代码时,THE Velotype 应自动包含存储的认证
3. IF 存储的凭据过期或无效,THEN THE Velotype 应提示用户重新认证
4. WHERE 认证失败,THE Velotype 应重试最多 3 次然后显示错误

### 需求 5: 执行输出显示选项

**用户故事:** 作为用户,我希望灵活查看代码执行结果,这样我可以为我的工作流选择最合适的界面。

#### 验收标准

1. WHERE 用户偏好外部终端,THE Velotype 应提供将执行结果发送到外部终端的选项
2. WHERE 用户偏好内置显示,THE Velotype 应在编辑器内的拆分窗格或叠加中显示执行结果
3. WHEN 执行输出显示时,THE Velotype 应显示:代码片段、输入参数、输出结果、执行时间和任何错误消息
4. IF 输出超过 10,000 个字符,THE Velotype 应截断并提供"查看完整输出"选项

### 需求 6: 会话管理

**用户故事:** 作为用户,我希望执行会话维护状态,这样我可以运行多个相互依赖的相关代码片段。

#### 验收标准

1. WHEN 顺序执行多个代码块时,THE Velotype 应在执行之间维护执行上下文(变量、加载的模块)
2. WHERE 代码块显式重置上下文,THE Velotype 应清除所有存储的状态
3. IF 执行失败,THE Velotype 应为后续执行保留上下文,除非显式重置
4. THE 上下文持久化持续时间应可配置:仅会话、文档打开期间或持久化到重启

### 需求 7: 配置持久化

**用户故事:** 作为用户,我希望我的服务器配置在会话之间持久化,这样我无需每次使用应用程序时都重新配置。

#### 验收标准

1. WHEN 服务器配置被保存,THE Velotype 应将其持久化到持久存储
2. WHEN 应用程序重启时,THE Velotype 应加载先前保存的服务器配置
3. WHERE 配置已被删除,THE Velotype 应从持久存储中移除它
4. THE 配置存储应支持来自多个文档的并发访问

### 需求 8: 连接池管理

**用户故事:** 作为用户,我希望连接高效重用,这样多个代码执行不会各自创建新连接。

#### 验收标准

1. WHEN 多个代码执行指向同一服务器时,THE Velotype 应从池中重用现有连接
2. WHERE 连接变旧或无效时,THE Velotype 应建立新连接
3. WHEN 无执行进行时,THE Velotype 应为可配置的持续时间维护空闲连接
4. THE 连接池应限制每个服务器的最大并发连接数

### 需求 9: 上下文隔离

**用户故事:** 作为用户,我希望执行上下文在文档之间隔离,这样一个文档的代码不会干扰另一个。

#### 验收标准

1. WHEN 从不同文档执行代码块时,THE Velotype 应维护单独的执行上下文
2. WHERE 文档指定自定义上下文 ID,THE Velotype 应使用该 ID 进行上下文隔离
3. IF 未指定自定义上下文 ID,THE Velotype 应从文档路径推导上下文 ID
4. CONTEXT 数据从一个文档不应可被另一个文档访问

### 需求 10: 超时配置

**用户故事:** 作为用户,我希望可配置的执行超时,这样长时间运行的代码不会无限期挂起。

#### 验收标准

1. WHEN 代码执行开始时,THE Velotype 应应用服务器配置中配置的超时
2. WHERE 未指定超时,THE Velotype 应使用默认 30 秒超时
3. IF 执行超过配置的超时,THE Velotype 应取消执行并返回超时错误
4. THE 超时应可每服务器和每文档配置

### 需求 11: 安全凭据存储

**用户故事:** 作为用户,我希望我的认证凭据安全存储,这样它们不会暴露给其他应用程序或用户。

#### 验收标准

1. WHEN 凭据被存储,THE Velotype 应在持久存储前加密它们
2. WHERE 凭据被检索,THE Velotype 应仅在内存中解密它们
3. THE 应用程序不应以任何形式记录或显示凭据
4. CREDENTIALS 应通过存储键隔离,不能在不同服务器配置之间访问

### 需求 12: 错误处理和恢复

**用户故事:** 作为用户,我希望强大的错误处理,这样执行失败不会崩溃应用程序。

#### 验收标准

1. WHEN 远程执行失败,THE Velotype 应捕获错误并显示用���友好的消息
2. IF 连接池遇到错误,THE Velotype 应在失败前重试操作
3. WHERE 发生不可恢复的错误,THE Velotype 应清理资源并返回空闲状态
4. ALL 错误应包括上下文信息(文档路径、代码语言、服务器配置)
